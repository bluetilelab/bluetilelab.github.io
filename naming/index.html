    Naming

    C++의 명명 규칙을 따르는 것은 코드 가독성, 유지보수성, 협업, 모범 사례 준수 등 여러 가지 이유로 필수적입니다. 명명 규칙을 잘 정의하면 본인뿐만 아니라 코드를 읽거나 함께 작업해야 하는 다른 개발자도 코드를 더 체계적이고 일관성 있으며 이해하기 쉽게 만들 수 있습니다.

    Function

    General

    일반적인 경우, PascalCase 를 사용합니다. 

    OpenFile()
Terminate()

    Inline or short

    인라인 함수 또는 길이가 짧은 함수의 경우, 예외적으로 snake_case 를 허용합니다.

    inline int sum_numbers(int x, int y) {
    return x + y;
};

class MyClass {
public:
    ...
    int type() const { return type_; }
    ...
};

    Info

    Qt Coding Style에 따르면 함수 이름은 camelCase 를 따라야 합니다. Qt와 같은 OpenCV, ChartDirector 등 크로스플랫폼 라이브러리들이 또한 camelCase 를 따르고 있습니다.

    반면 구글 C++ 스타일 가이드 및 일반적인 Win32 C++ 개발 문화에서는 PascalCase 가 주류입니다.

    우리 BTL은 다음과 같은 이유로 일반 함수에서 PascalCase 를 사용하기로 결정하였습니다.

    • PascalCase가 익숙합니다.
    • 라이브러리 API 함수들과 직접 작성한 함수들을 casing 으로 구분할 수 있습니다.

    Variable

    Common

    모든 변수 타입에 공통으로 적용하는 규칙

    우리는 헝가리안 표기법 (Hungarian Notation) 을 권장하지 않습니다.

    Visual Studio 와 같은 IDE 들은 변수의 타입을 구분할 수 있는 충분한 부가 정보를 제공하고 있습니다.

    • 색상으로 syntax highlighting
      • 변수 타입 "색상 구분 가능"
      • 이게 로컬 변수인지, 클래스 변수인지, 글로벌 변수인지 1
    • 마우스 hovering 시 "아주 자세한" 정보 표시
    • 코드 작성 중 잘못된 타입 할당에 대하여 빌드없이도 "즉시 에러 표시" 2
    • Inline hint
      • 마우스 hovering 없이도 타입 정보 확인
      • auto 타입 추론
      • 함수 파라미터 타입 정보
    공통 변수 명명법
    int type;   // O (1)   
int n_type; // X

size_t num_entries; // O (2)                    
size_t m_nEntries;  // X

QString username;   // O
QString qsUserName; // X (3)

QWidget* main_widget;       // O 
QWidget* pControlWidget;    // X (4)
    1. 때로는 type 이 문자열, 열거형일 수도 있습니다. 확인을 위해 우리는 조심스럽게 마우스 올리고 변수의 타입을 확인합니다.
    2. 'num' 사용으로 '개수'라는 의미 전달합니다. 'num_of', 'number_of_" 등도 가능합니다.
    3. Qt 타입마다 모두 'q' 를 붙여야 합니다. 사양합니다.
    4. 포인터에 'p'를 붙이는 관용이 있습니다. 사양합니다.

    Class Member

    클래스 멤버 변수는 snake_case 를 사용합니다.

    클래스 멤버 변수
    class MyClass {
private:
    QString name;
    QQueue fifo_queue;
};

    Note

    구글 C++ 스타일 가이드 및 다른 가이드에서는 클래스 멤버 변수, 특히 private 을 구분짓기 위해 _ (underscore) 를 head 또는 tail 에 붙이는 것이 관례입니다. 그러나 우리는 다음과 같은 이유로 _ 를 사용하지 않습니다.

    • IDE 가 제공하는 폰트 색상 및 볼드체를 통해 변수 타입을 구별할 수 있습니다.
    • 외부에서 접근 권한이 없는 private 변수는 IDE 가 자동 완성 리스트에 포함하지 않습니다.
    • 미관상 보기 싫습니다.

    Local

    로컬 변수는 snake_case 를 사용합니다.

    Note

    로컬 변수는 전역 변수 또는 클래스 멤버 변수가 아닌 일반적으로 우리가 함수 scope 한정으로 임시 사용하는 변수를 일컫습니다.

    Namespace

    snake_case 를 사용합니다.

    Namespace
    namespace btl {

}

namespace enum_helper {

}

    Enum

    타입명은 PascalCase, 멤버명은 SNAKE_CASE 를 사용합니다.

    Enum
    enum class Code {
    INVALID = -1,
    EC_INSP_FILE_OPEN_FAIL,
    IC_ROI_TIME_ELAPSED,     
};

enum class ResultTypes {
    INVALID = -1,
    OK, 
    NG, 
    ERROR,
};

    Macro

    SNAKE_CASE 를 사용합니다.

    Macro
    #define EXPANDING QSizePolicy(QSizePolicy::Expanding, QSizePolicy::Expanding)
#define HORIZONTAL_SPACER new QSpacerItem(40, 20, QSizePolicy::Expanding, QSizePolicy::Minimum)

    Info

    구글 C++ 스타일 가이드 는 매크로를 사용하면 안 되는 이유를 설명합니다. 그러나 저희는 반복적인 코드 작성을 피하기 위해 문자열 치환과 같은 단순한 용도를 허용합니다.

    1. 경험상, 기본 다크가 가장 무난합니다. ↩

    2. IDE 가 표시한 경고 및 에러는 "절대" 무시하지 않습니다. 무시해도 상관없다면 그 근거를 제시하십시오. ↩
